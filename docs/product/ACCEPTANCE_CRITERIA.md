# 验收标准

本文档用于描述企业多源知识与数据智能协作平台在当前阶段和后续阶段的可测试验收条件。

## 1. 前端聊天工作台

- 当用户在前端输入自然语言问题并提交时，系统应调用 `POST /api/chat`。
- 当后端成功返回答案时，前端应在消息列表中展示助手回复。
- 当后端返回异常时，前端应展示错误信息，而不是静默失败。
- 当用户设置 `thread_id` 和 `user_id` 时，前端应随请求发送对应参数。
- 当请求执行中时，前端应显示加载状态，并临时禁用相关按钮。

## 2. 后端 API

- 当访问 `GET /api/health` 时，系统应返回服务健康状态。
- 当调用 `POST /api/chat` 时，系统应将问题交给 LangGraph 多代理系统处理。
- 当调用 `GET /api/state` 时，系统应返回指定线程的状态快照。
- 当调用 `POST /api/review/approve` 时，系统应提交审批通过并恢复线程。
- 当调用 `POST /api/review/reject` 时，系统应提交审批拒绝并阻止危险操作。
- 当调用 `POST /api/graph/build` 时，系统应执行图谱 dry-run 或导入流程，并返回日志。

## 3. Supervisor 路由

- 当用户提出普通解释类问题时，系统应优先路由到 `chat`。
- 当用户提出结构化数据查询或写库问题时，系统应路由到 `sqler`。
- 当用户提出图表生成问题时，系统应先由 `sqler` 获取数据，再由 `coder` 生成图表。
- 当用户提出关系型知识问题时，系统应路由到 `graph_kg`。
- 当用户提出文档语义检索问题时，系统应路由到 `vec_kg`。
- 当 worker 已经给出有效答案后，系统应结束当前任务，避免重复循环。

## 4. GraphRAG / graph_kg

- 当用户询问“都有哪些公司在我的数据库中？”时，系统应返回 Neo4j 中的 `Company` 节点列表。
- 当用户询问“华为技术有限公司与哪些教育机构建立了合作？”时，系统应返回对应教育或研究机构。
- 当问题命中高频确定性图谱查询时，系统应使用手写 Cypher，而不是让 LLM 临时生成 Cypher。
- 当问题未命中确定性分支时，系统可使用 `GraphCypherQAChain` 生成 Cypher 并查询 Neo4j。
- 当 Cypher 生成失败时，系统后续应返回可读错误或降级方案。

## 5. Vector RAG / vec_kg

- 当用户提出适合文档语义检索的问题时，系统应调用向量数据库检索相关上下文。
- 当向量库不可用时，系统应返回明确错误来源。
- 后续当加入 RAG A/B 测试后，系统应记录不同切块、检索、重排序策略的效果指标。

## 6. 图谱构建与治理

- 当执行 `python build_graph_offline.py --dry-run` 时，系统不得写入 Neo4j。
- 当 dry-run 完成时，系统应输出实体数量、关系数量和评测命中率。
- 当执行正式导入时，系统应将通过 schema guard 的实体关系写入 Neo4j。
- 当执行 `--reset` 时，系统应清理旧图并重建干净图。
- 当构图逻辑变更后，`kg_eval.py` 的核心期望关系不应退化。

## 7. 人机审批

- 当用户请求删除、修改或写库操作时，系统应进入 HITL 审批流程。
- 当审批通过时，系统应继续执行原任务。
- 当审批拒绝时，系统应中止危险操作并返回说明。
- 当未来加入审批日志时，每次审批应记录用户、线程、操作、结果和时间。

## 8. 记忆机制

- 当前系统应支持基于 `thread_id` 的短期上下文恢复。
- 当前系统应支持基于 `user_id` 的长期记忆原型。
- 后续迁移到持久化存储后，服务重启不应丢失 checkpoint 和长期记忆。
- 后续记忆提取层上线后，系统不应把所有对话无差别写入长期记忆。

## 9. 前端响应式体验

- 在桌面浏览器中，前端应展示多栏工作台。
- 在窄屏设备中，前端应自动切换为单栏布局。
- 审批、状态查看、图谱构建等操作应有明确按钮和执行反馈。
