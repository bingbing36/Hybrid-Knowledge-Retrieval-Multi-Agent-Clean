# 用户故事

本文档基于“企业多源知识与数据智能协作平台”的能力定位，描述不同角色在系统中的核心使用诉求。

## 1. 企业业务用户

- 作为企业业务用户，我希望用自然语言查询不同系统里的信息，这样我不需要理解 SQL、Cypher 或向量检索细节。
- 作为企业业务用户，我希望系统能自动判断问题应该查数据库、文档还是图谱，这样我可以专注于业务问题本身。
- 作为企业业务用户，我希望答案能说明来源或依据，这样我可以判断结果是否可信。
- 作为企业业务用户，我希望系统能记住常用查询偏好，这样后续分析可以更贴合我的工作习惯。

## 2. 数据分析人员

- 作为数据分析人员，我希望用自然语言触发结构化数据查询，这样我可以减少临时写 SQL 的成本。
- 作为数据分析人员，我希望系统能把查询结果转成图表，这样我可以快速完成数据解释和汇报。
- 作为数据分析人员，我希望复杂任务能被拆成“查数、分析、画图、总结”多个步骤，这样任务过程更可控。
- 作为数据分析人员，我希望后续能对不同检索策略做 A/B 测试，这样我可以量化优化效果。

## 3. 知识库管理员

- 作为知识库管理员，我希望先 dry-run 查看文档抽取出的实体和关系，这样我可以在写入图数据库前检查质量。
- 作为知识库管理员，我希望系统限制实体类型和关系类型，这样图谱不会被普通短语污染。
- 作为知识库管理员，我希望系统能合并简称、英文名和规范名，这样同一实体不会被重复创建。
- 作为知识库管理员，我希望看到构图评测结果，这样我可以判断本次图谱构建是否退化。
- 作为知识库管理员，我希望重建图谱前有明确确认，这样我可以避免误删已有数据。

## 4. 安全审批人员

- 作为安全审批人员，我希望删除、修改、写库等危险操作自动进入人工审批，这样可以避免 AI 误操作。
- 作为安全审批人员，我希望能在前端点击批准或拒绝，这样审批流程不依赖命令行。
- 作为安全审批人员，我希望后续能查看审批日志，这样可以追踪谁在什么时候批准了什么操作。
- 作为安全审批人员，我希望系统能清楚解释被拒绝操作的原因，这样用户知道下一步该怎么做。

## 5. AI 工程学习者 / 面试展示者

- 作为学习者，我希望看到一个完整的多代理工程项目，而不仅是 Notebook 片段。
- 作为学习者，我希望理解 ReAct、Plan-and-Execute、Workflow 的区别，这样我可以知道不同 Agent 架构适合什么任务。
- 作为面试展示者，我希望能演示 SQL 查询、GraphRAG、Vector RAG、图表生成、HITL 和记忆机制，这样项目能力更完整。
- 作为面试展示者，我希望能展示 ERROR_LOG、PRD、Future Roadmap 和评测指标，这样可以体现真实工程迭代过程。

## 6. 开发者 / 集成方

- 作为开发者，我希望通过 FastAPI 调用多代理能力，这样可以把系统接入其他业务平台。
- 作为开发者，我希望 API 有清晰的错误分类，这样可以快速定位是模型、Neo4j、Milvus 还是业务逻辑问题。
- 作为开发者，我希望前端和后端解耦，这样后续可以单独升级 UI 或 Agent 服务。
- 作为开发者，我希望后续支持 SSE 或 WebSocket，这样可以展示 Agent 的实时执行过程。
