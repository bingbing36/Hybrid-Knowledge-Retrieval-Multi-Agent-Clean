# 版本规划

本文档用于记录项目从当前教学型原型，逐步演进为“企业多源知识与数据智能协作平台”的路线。这里的重点不是绑定某一份演示数据，而是围绕“自然语言访问企业知识、数据库和分析工具”的能力逐步升级。

## MVP：前后端可用工作台

目标是先把项目从命令行脚本升级成可演示、可联调、可复用的产品原型。

- 提供 FastAPI 后端接口，封装对话、状态查看、人机审批和图谱构建能力。
- 提供 React/Vite 前端工作台，支持输入问题、查看回答、选择示例指令、触发审批操作。
- 保留命令行入口，方便教学、调试和复现 Notebook 中的调用方式。
- 将 chat、sqler、coder、graph_kg、vec_kg 的多代理能力通过统一入口对外暴露。
- 在 README 中补充前后端启动方式、典型问题和验证步骤。

## V1：事件流与执行轨迹

目标是让用户不只看到最终答案，也能看见系统“为什么这么回答”。

- 增加 SSE 或 WebSocket 流式输出，让前端可以实时展示模型响应。
- 在前端展示 supervisor 的路由结果，例如本轮问题被分配给 graph_kg、vec_kg、sqler 或 coder。
- 展示关键工具调用过程，例如 SQL 查询、图谱查询、向量检索、图表生成。
- 展示中间消息和最终回答，方便定位“答非所问”“路由错误”“工具没执行”等问题。
- 对图表生成结果增加前端预览，避免用户只看到文本占位符。

## V2：生产级记忆与审计

目标是把当前适合开发验证的内存型记忆，升级成可持久化、可追踪、可恢复的企业级记忆层。

- 将短期记忆从 `MemorySaver` 扩展为 SQLite 或 PostgreSQL checkpoint。
- 将长期记忆从 `InMemoryStore` 扩展为 PostgreSQL、Redis 或向量数据库方案。
- 为不同用户、不同会话、不同业务空间设计隔离机制。
- 记录人机审批日志，包括用户输入、审批人、审批结果、执行时间和工具结果。
- 增加记忆管理能力，支持查看、清理、禁用或迁移用户记忆。

## V3：RAG 与 GraphRAG 效果优化

目标是从“能检索”升级到“可评测、可对比、可持续优化”。

- 建立固定评测集，覆盖事实问答、关系推理、数据库查询、图表生成和危险操作审批。
- 对文本切块策略做 A/B 测试，例如固定长度切块、语义切块、标题层级切块。
- 对混合检索策略做对比，例如向量检索、关键词检索、图谱检索、多路召回。
- 引入召回重排序能力，对不同 reranker 方案进行评测。
- 增加 Query 优化能力，例如查询改写、多查询扩展、HyDE 或意图补全。
- 用召回率、准确率、答案满意度、响应时延等指标量化优化收益。

## V4：Agent 架构演进

目标是让项目不只展示 ReAct Agent，也能体现不同 Agent 架构适合的任务边界。

- 保留当前 ReAct 风格代理，继续承担工具调用和即时问答任务。
- 增加 Plan-and-Execute Agent，用于多步骤分析、复杂数据任务和跨知识库推理。
- 增加 Workflow 模式，用于审批、删除、修改等流程固定且安全要求高的任务。
- 对比 ReAct、Plan-and-Execute、Workflow 的适用场景、优缺点和失败模式。
- 将路由效果纳入评测，持续优化 supervisor 的任务分配准确率。

## V5：企业化落地能力

目标是从个人演示项目走向企业内部可试点系统。

- 增加登录认证、角色权限、数据空间隔离和操作权限控制。
- 支持多用户、多会话、多知识库管理。
- 增加图谱可视化页面，展示实体、关系、来源文档和构建质量。
- 增加运行监控，包括模型调用、数据库连接、检索耗时、错误类型和成本统计。
- 补充 Docker、部署配置、环境变量说明和生产启动方案。
- 为核心能力沉淀演示脚本和面试讲解材料，方便展示项目价值。

## 发布原则

- 先验证能力闭环，再追求界面完整。
- 每次新增核心逻辑，都同步检查 README、ERROR_LOG 和相关产品文档是否需要更新。
- 涉及删除、修改数据库等高风险操作，必须保留人机审批。
- RAG、GraphRAG、意图识别和 Agent 架构优化必须有评测数据支撑，避免只凭主观感觉判断效果。
- 数据可以替换，但能力定位要稳定：让用户用自然语言调度企业知识、结构化数据和分析工具。
