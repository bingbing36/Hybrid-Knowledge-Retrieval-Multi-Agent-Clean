# 风险与缓解方案

本文档记录企业多源知识与数据智能协作平台在产品化、工程化和企业落地中的主要风险。

## 1. 意图识别风险

### 风险

- Supervisor 将问题路由到错误 Agent。
- 高频问题被过宽规则误判。
- LLM 路由输出格式不稳定。

### 缓解方案

- 高频确定性问题使用精确规则。
- 建立意图识别评测集。
- 使用本地解析和白名单校验 LLM 路由结果。
- 对 worker 已回答场景及时结束，避免循环。

## 2. 图谱构建风险

### 风险

- 文档抽取出的实体边界不准确。
- 关系类型过于开放，导致图谱难以查询。
- 旧脏图未清理，新逻辑仍被历史数据污染。
- LLMGraphTransformer 输出不稳定。

### 缓解方案

- 使用 `kg_schema.py` 约束实体类型和关系类型。
- 使用 `entity_resolution.py` 做实体归一化。
- 使用 `kg_eval.py` 做构图回归评测。
- 默认不将 LLMGraphTransformer 开放抽取结果写入主图。
- 使用 `--dry-run` 先预览，必要时用 `--reset` 清理旧图。

## 3. GraphRAG 查询风险

### 风险

- LLM 生成错误 Cypher。
- 图谱 schema 变化后旧 prompt 失效。
- 高频问题每次都依赖 LLM 生成查询，稳定性不足。

### 缓解方案

- 高频确定性问题使用手写 Cypher。
- 对关键图谱问答建立回归样例。
- 保留 LLM-to-Cypher 作为开放问题能力，但加异常处理和降级方案。

## 4. Vector RAG 风险

### 风险

- 文本切块不合理导致召回失败。
- 向量检索召回相关但不精确的片段。
- 没有 rerank 时答案容易基于弱相关上下文生成。
- 向量库连接失败影响问答能力。

### 缓解方案

- 做切块策略 A/B 测试。
- 引入混合检索。
- 引入 reranker。
- 建立 Recall@K、Precision@K、MRR 等检索指标。
- 对 Milvus 连接错误做明确提示。

## 5. 工具执行风险

### 风险

- SQL 写操作误删或误改数据。
- Python 代码执行产生错误文件或不安全操作。
- 图表生成失败但模型误称已经完成。

### 缓解方案

- 删除、修改、写库操作进入 HITL。
- `sqler` 只负责取数，不声称完成图表。
- `coder` 负责代码执行和图表落地。
- 对代码执行环境做限制和日志记录。

## 6. 记忆风险

### 风险

- 当前 `MemorySaver` 和 `InMemoryStore` 不适合生产。
- 长期记忆可能保存无价值内容。
- 多 Agent 共享过多记忆导致上下文混乱。
- 敏感信息可能被写入长期记忆。

### 缓解方案

- 将 checkpoint 迁移到 SQLite / PostgreSQL。
- 长期记忆采用 PostgreSQL metadata + 向量库。
- 增加记忆提取层，只保存高价值记忆。
- 为不同 Agent 设计不同记忆注入策略。
- 支持记忆查看、修改、删除和审计。

## 7. 前后端工程风险

### 风险

- 当前前端为同步请求，复杂任务等待时间较长。
- API 暂无鉴权，不适合公网暴露。
- 前端状态与 LangGraph 线程状态可能不同步。

### 缓解方案

- 后续加入 SSE / WebSocket。
- 增加用户登录、权限和会话管理。
- 前端增加 pending review、running、error 等状态提示。
- 后端统一错误分类。

## 8. 模型与外部服务风险

### 风险

- DeepSeek 或 Qwen 余额不足。
- Neo4j / Milvus 连接失败。
- 模型响应格式变化导致解析失败。

### 缓解方案

- 保留 provider 错误分类提示。
- 保留 Neo4j / Milvus 认证错误提示。
- 关键路径增加确定性兜底。
- 对模型输出做本地校验和异常处理。

## 9. 企业落地风险

### 风险

- 不同行业数据 schema 差异大。
- 企业对权限、审计、数据安全要求高。
- 业务用户可能不理解 Agent 边界。

### 缓解方案

- 将 demo 数据视为样例，底层能力保持行业无关。
- 增加 schema 配置能力。
- 增加权限、审计、日志和数据隔离。
- 在前端增加 Agent 能力边界提示和操作反馈。
