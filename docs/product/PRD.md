# PRD：企业多源知识与数据智能协作平台

## 1. 产品定位

本项目定位为一个面向企业内部复杂信息查询与任务执行的多代理智能协作平台。

它不绑定某一类固定业务数据，而是提供一套通用能力：通过自然语言统一调度结构化数据库、文档知识库、图数据库、向量数据库、代码执行工具、长短期记忆和人工审批流程，帮助用户完成查询、检索、推理、分析、可视化和安全操作。

一句话定位：

> 面向企业多源知识与数据场景的多代理智能工作台，让业务用户用自然语言完成跨系统查询、混合知识检索、关系推理、数据分析、图表生成和安全任务执行。

## 2. 背景与问题

企业内部的信息通常分散在不同系统中：

- 结构化业务数据在 SQL 数据库中。
- 非结构化知识在文档、制度、产品说明、项目资料中。
- 关系型知识分散在公司、人员、产品、客户、供应商、合作伙伴、机构、技术之间。
- 历史对话、用户偏好、审批记录和长期任务背景缺乏统一记忆管理。
- 一线业务人员往往不会写 SQL、Cypher 或 Python，也不熟悉向量检索和图谱查询。
- 删除、修改、写库、执行代码等高风险操作不能完全交给 AI 自动执行。

传统单一 RAG 或单一 Chatbot 很难覆盖这些场景。企业真正需要的是一个能理解意图、选择工具、调用知识库、执行任务并在关键节点请求人工确认的智能协作平台。

## 3. 产品目标

### 3.1 核心目标

- 支持用户用自然语言访问多源知识与数据。
- 通过 Supervisor 自动识别意图并路由到合适 Agent。
- 整合 SQL 查询、GraphRAG、Vector RAG、代码执行、人工审批和记忆机制。
- 提供前后端分离工作台，让系统从“脚本能力”升级为“产品雏形”。
- 建立可评测、可迭代、可扩展的多代理工程化框架。

### 3.2 能力目标

- 多源接入：SQL、Neo4j、向量库、文档、长期记忆。
- 多代理协作：`chat / sqler / coder / graph_kg / vec_kg`。
- 混合检索推理：结构化查询、语义检索、图谱关系推理、多路知识融合。
- 安全执行：危险操作进入 human-in-the-loop 审批。
- 可视化交互：通过 React/Vite 前端提供聊天、状态、审批、图谱构建入口。
- 可评测优化：通过评测集、A/B 测试和指标体系持续优化效果。

## 4. 非目标

当前阶段不解决以下问题：

- 不做完整企业级权限系统。
- 不直接替换为大型现成 GraphRAG / LightRAG 框架。
- 不把示例销售数据作为唯一行业定位。
- 不承诺当前 `MemorySaver` / `InMemoryStore` 达到生产持久化要求。
- 不在 MVP 阶段实现完整 Agent Trace、图谱大屏和多租户管理。

## 5. 目标用户

### 5.1 企业业务用户

希望通过自然语言查询业务数据、知识文档、关系图谱和历史信息，不需要掌握 SQL、Cypher 或 Python。

### 5.2 数据分析人员

希望让系统自动完成查数、分析、画图和解释结果，提高临时分析和经营洞察效率。

### 5.3 知识库管理员

希望构建、清理、评估和维护企业知识库、向量库与图数据库。

### 5.4 安全审批人员

希望对删除、修改、写库、代码执行等高风险操作进行人工确认和审计。

### 5.5 AI 工程学习者 / 面试展示者

希望通过一个完整项目展示多代理、RAG、GraphRAG、HITL、记忆、前后端工程化和评测优化能力。

## 6. 典型使用场景

### 6.1 多源知识问答

用户提问：

```text
某个产品的关键技术、合作机构和市场布局是什么？
```

系统根据意图选择图谱查询、向量检索或多路检索，返回结构化答案。

### 6.2 图谱关系推理

用户提问：

```text
华为技术有限公司与哪些教育机构建立了合作？
```

系统通过 `graph_kg` 查询 Neo4j 中的实体关系，返回合作机构。

### 6.3 结构化数据分析

用户提问：

```text
帮我分析最近销售额最高的记录，并生成柱状图。
```

系统先由 `sqler` 查询结构化数据，再由 `coder` 执行 Python 生成图表。

### 6.4 普通知识问答

用户提问：

```text
什么是机器学习？
```

系统路由到 `chat`，直接进行通用问答。

### 6.5 安全写操作审批

用户提问：

```text
帮我删除某条业务记录。
```

系统识别为危险写库操作，进入 HITL 审批。审批通过才继续执行，审批拒绝则中止操作。

### 6.6 图谱构建与评测

知识库管理员触发图谱 dry-run，系统展示实体数量、关系数量、评测命中率和抽取预览，再决定是否导入 Neo4j。

## 7. 核心功能需求

### 7.1 Supervisor 意图识别与路由

- 系统应识别用户问题类型。
- 系统应将问题路由到最合适的 Agent。
- 高频确定性问题应优先使用规则分支，避免 LLM 生成不稳定查询。
- 复杂模糊问题可继续使用 LLM 路由。

### 7.2 多 Agent 协作

- `chat`：处理通用解释、闲聊和基础问答。
- `sqler`：处理结构化数据库查询、修改和删除。
- `coder`：处理 Python 执行、数据分析、图表生成。
- `graph_kg`：处理图数据库关系查询和 GraphRAG 问答。
- `vec_kg`：处理文档向量检索和语义问答。

### 7.3 混合知识检索

- 系统应支持图谱检索、向量检索、SQL 查询和长期记忆召回。
- 系统后续应支持多路检索结果融合。
- 系统应支持 RAG 策略 A/B 测试，包括切块、混合检索、重排序和 Query 优化。

### 7.4 图谱构建与治理

- 系统应支持从文档中抽取实体和关系。
- 系统应通过 `kg_schema.py` 限制实体类型和关系类型。
- 系统应通过 `entity_resolution.py` 合并简称、别名和规范名。
- 系统应通过 `kg_eval.py` 对构图结果进行回归评测。
- 系统应支持 `--dry-run` 预览抽取结果。
- 系统应支持 `--reset` 清理旧脏图并重建干净图。

### 7.5 人机审批

- 删除、修改、写库等危险操作必须进入人工审批。
- 审批通过后继续执行。
- 审批拒绝后中止操作，并返回明确说明。
- 后续应持久化审批记录，支持审计。

### 7.6 记忆机制

- 当前支持短期线程记忆和长期内存记忆。
- 后续应将短期 checkpoint 迁移到 SQLite / PostgreSQL。
- 后续应将长期记忆迁移到 PostgreSQL + 向量库。
- 后续应增加记忆提取层，避免无价值信息污染长期记忆。

### 7.7 前后端工作台

- 后端通过 FastAPI 暴露核心能力。
- 前端通过 React/Vite 提供交互界面。
- 前端应支持聊天、线程参数、快捷问题、人工审批、状态查看、图谱构建。
- 后续应支持 SSE / WebSocket 流式输出和 Agent Trace。

## 8. API 需求

当前 API：

- `GET /api/health`：检查服务健康状态。
- `POST /api/chat`：提交问题并返回 Agent 答案。
- `GET /api/state`：查看指定线程状态。
- `POST /api/review/approve`：批准待审批任务。
- `POST /api/review/reject`：拒绝待审批任务。
- `POST /api/graph/build`：触发图谱构建或 dry-run。

后续 API：

- `GET /api/events`：SSE 事件流。
- `WS /api/ws/chat`：WebSocket 双向对话。
- `GET /api/evaluations`：查看评测结果。
- `POST /api/evaluations/run`：触发评测任务。
- `GET /api/memories`：查看长期记忆。
- `DELETE /api/memories/{id}`：删除指定记忆。

## 9. 成功指标

### 9.1 产品指标

- 用户问题提交成功率。
- API 调用成功率。
- 前端任务完成率。
- 用户满意度。

### 9.2 Agent 指标

- Supervisor 路由准确率。
- 工具调用成功率。
- `graph_kg` 问答准确率。
- `vec_kg` 检索 Recall@K / Precision@K。
- `coder` 图表生成成功率。
- HITL 危险操作拦截准确率。

### 9.3 工程指标

- 端到端响应延迟。
- 模型调用次数。
- token 成本。
- Neo4j 查询延迟。
- Milvus 查询延迟。
- 回归测试通过率。

## 10. 关键风险

- LLM 路由错误导致 Agent 选择不准。
- LLM 生成 Cypher 错误导致图谱查询失败。
- 文档抽取不准导致脏图进入 Neo4j。
- 向量检索召回不准导致答案缺乏依据。
- 记忆长期累积后可能出现记忆污染。
- 危险操作如果绕过审批，会带来数据安全风险。
- 当前 API 暂无鉴权，不适合直接暴露公网。

## 11. 风险缓解

- 高频稳定问题使用确定性规则和手写查询。
- 图谱构建使用 schema guard、实体归一化和评测。
- 危险操作强制进入 HITL。
- 重要错误写入 `ERROR_LOG.md`。
- 后续建立 RAG、路由、图谱、记忆和 HITL 的评测集。
- 生产部署前增加鉴权、权限、审计和持久化存储。

## 12. 版本规划

### MVP：前后端可用工作台

- FastAPI API 封装。
- React/Vite 聊天界面。
- 人工审批按钮。
- 状态查看。
- 图谱构建入口。

### V1：事件流与 Agent Trace

- 增加 SSE / WebSocket。
- 展示 Agent 中间过程。
- 展示工具调用、路由和执行状态。

### V2：生产级记忆

- SQLite checkpoint。
- PostgreSQL checkpoint。
- PostgreSQL + Milvus 长期记忆。
- 审批日志持久化。

### V3：RAG 优化与评测

- 文本切块 A/B 测试。
- 混合检索。
- 召回重排序。
- Query rewrite / Multi-query / HyDE。
- 检索与答案质量评测面板。

### V4：Agent 架构演进

- 加入 Plan-and-Execute Agent。
- 加入固定 Workflow。
- 对比 ReAct、Plan-and-Execute、Workflow 的适用任务。

### V5：企业化能力

- 登录与权限。
- 多用户、多会话管理。
- 记忆管理 UI。
- 图谱可视化。
- 部署与监控。
